
## LLMNR y NBT-NS

[Enlace-Local Multicast Resolución](https://datatracker.ietf.org/doc/html/rfc4795) de [Nombres](https://datatracker.ietf.org/doc/html/rfc4795) (LLMNR) y [NetBIOS Name Service](https://docs.microsoft.com/en-us/previous-versions/windows/it-pro/windows-2000-server/cc940063\(v=technet.10\)?redirectedfrom=MSDN) (NBT-NS) son componentes de Microsoft Windows que sirven como métodos alternativos de identificación de host que se pueden utilizar cuando DNS falla. Si una máquina intenta resolver un huésped, pero la resolución DNS falla, normalmente, la máquina intentará pedir a todas las demás máquinas de la red local la dirección de host correcta a través de LLMNR. LLMNR se basa en el formato del Sistema de Nombres de Dominio (DNS) y permite a los hosts en el mismo enlace local para realizar la resolución de nombres para otros anfitriones. Utiliza el puerto `5355`sobre UDP de forma nativa. Si LLMNR falla, se utilizará el NBT-NS. NBT-NS identifica los sistemas en una red local por su nombre de NetBIOS. NBT-NS utiliza puerto `137`sobre UDP.

## Mitigación

Mitre ATT&CK enumera esta técnica como [ID: T1557.001](https://attack.mitre.org/techniques/T1557/001), `Adversary-in-the-Middle: LLMNR/NBT-NS Poisoning and SMB Relay`.

Hay algunas maneras de mitigar este ataque. Para asegurarnos de que estos ataques de falsificación no son posibles, podemos desactivar LLMNR y NBT-NS. Como una palabra de precaución, siempre vale la pena probar lentamente un cambio significativo como este a su entorno cuidadosamente antes de desplegarlo completamente. Como probadores de penetración, podemos recomendar estos pasos de remediación, pero debemos comunicar claramente a nuestros clientes que deben probar estos cambios fuertemente para asegurar que la desactivación de ambos protocolos no rompa nada en la red.

Podemos deshabilitar LLMNR en la Política de Grupos yendo a la Configuración de Equipos ---- Templates Administrativas ---- Red ----- Cliente DNS y habilitando "Resolución de Nombres Multicast OFF".

NBT-NS no puede desactivarse a través de la política de grupo, sino que debe ser desactivada localmente en cada huésped. Podemos hacerlo abriendo `Network and Sharing Center`en el marco de la `Control Panel`, haciendo clic en `Change adapter settings`, haciendo clic derecho en el adaptador para ver sus propiedades, seleccionando `Internet Protocol Version 4 (TCP/IPv4)`, y haciendo clic en el `Properties`botón, luego haciendo clic en `Advanced`y seleccionan la `WINS`pestaña y finalmente selección `Disable NetBIOS over TCP/IP`.

Si bien no es posible desactivar NBT-NS directamente a través de GPO, podemos crear un script de PowerShell bajo la Configuración de la computadora ---Configuración de Windows --- Script (Startup/Shutdown) --- Startup con algo como lo siguiente:

```powershell

$regkey = "HKLM:SYSTEM\CurrentControlSet\services\NetBT\Parameters\Interfaces"
Get-ChildItem $regkey |foreach { Set-ItemProperty -Path "$regkey\$($_.pschildname)" -Name NetbiosOptions -Value 2 -Verbose}
```

En el Editor de Políticas de Grupo Local, tendremos que hacer doble clic en `Startup`, elija el `PowerShell Scripts`pestaña, y seleccione "Para este GPO, ejecute scripts en el siguiente orden" para `Run Windows PowerShell scripts first`, y luego haga clic en `Add`y elige el guión. Para que estos cambios ocurran, tendríamos que reiniciar el sistema de destino o reiniciar el adaptador de red.

Para forzar esto a todos los hosts en un dominio, podríamos crear un GPO usando `Group Policy Management`en el controlador de dominio y aloje el script en la cuota de SYSVOL en la carpeta de scripts y luego llámelo a través de su ruta de UNC.

Una vez que el GPO se aplica a las OU específicas y se reinician esos anfitriones, el script se ejecutará en el siguiente reinicio y deshabilitará NBT-NS, siempre que el script todavía exista en la parte de SYSVOL y sea accesible por el host a través de la red.

No siempre es posible desactivar LLMNR y NetBIOS, y por lo tanto necesitamos maneras de detectar este tipo de comportamiento de ataque. Una forma es utilizar el ataque contra los atacantes inyectando solicitudes de LLMNR y NBT-NS para huéspedes inexistentes a través de diferentes subredes y alertando si alguna de las respuestas recibe respuestas que serían indicativas de una solución de nombres atacante respuestas de resolución de nombres. Esta [entrada](https://www.praetorian.com/blog/a-simple-and-effective-way-to-detect-broadcast-name-resolution-poisoning-bnrp/) de [blog](https://www.praetorian.com/blog/a-simple-and-effective-way-to-detect-broadcast-name-resolution-poisoning-bnrp/) explica este método más en profundidad.
