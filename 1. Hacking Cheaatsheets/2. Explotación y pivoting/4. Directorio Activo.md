## explanation

Several vulnerabilities have been recently disclosed, namely:

- **MS-EFSRPC** – AKA PetitPotam
- Credential Relaying abusing the AD CS role

Any attacker with internal network access, such as a phished client or a malicious planted device in the network, can take over the entire Active Directory domain without any initial credentials. Domain Controllers and AD CS is vulnerable to this attack currently in the default configuration.

An attacker can trigger a Domain Controller using PetitPotam to NTLM relay credentials to a host of choice. The Domain Controller’s NTLM Credentials can then be relayed to the Active Directory Certificate Services (AD CS) Web Enrollment pages, and a DC certificate can be enrolled. This certificate can then be used to request a TGT (Ticket Granting Ticket) and compromise the entire domain through Pass-The-Ticket.

AD CS (Active Directory Certificate Services) is particularly interesting as it offers role services that by default accept NTLM-based authentication. These services specifically include **Certificate Authority Web Enrollment** and the **Certificate Enrollment Web Service**.

## Conclusion

These two bugs chained together allow for instant escalation from low privilege user to domain administrator in a matter of minutes. This is a devastating bug that at the time of writing will work on a fully patched domain controller and active directory certificate services setup.

## Mitigations

- **Remove the listed role services** if not justified by a business need. In most cases, the affected services are replaceable by other API/interfaces such as the built-in RPC interfaces.
- **Restrict/disable inbound NTLM authentication** to the server running the role service by setting the policy “Network security: Restrict NTLM: Incoming NTLM traffic”.
- **Disable/remove the NTLM provider** in the Internet Information Services (IIS) running the selected role services.

# Protocolos de AD

## SMB

SMB es un protocolo de red que permite compartir archivos, impresoras y otros recursos en una red de área local (LAN). Fue desarrollado por Microsoft y se utiliza principalmente en entornos Windows. SMB permite a los clientes conectarse a un servidor y acceder a recursos compartidos, como carpetas y archivos.

Aquí hay un resumen de cómo funciona SMB:

1. Un cliente (por ejemplo, un equipo con Windows) se conecta a un servidor que comparte recursos a través de SMB.
2. El cliente envía una solicitud de conexión al servidor, que incluye el nombre del recurso compartido y las credenciales de autenticación (nombre de usuario y contraseña).
3. El servidor verifica las credenciales y, si son válidas, devuelve un identificador de sesión al cliente.
4. El cliente utiliza el identificador de sesión para acceder al recurso compartido y realizar operaciones, como leer o escribir archivos.

## SAMBA

SAMBA es una implementación de código abierto del protocolo SMB. Fue desarrollada para permitir que los sistemas Unix y Linux se conecten a recursos compartidos de Windows y viceversa. SAMBA actúa como un servidor SMB y permite a los clientes Windows conectarse a recursos compartidos en un servidor Unix o Linux.

Aquí hay un resumen de cómo funciona SAMBA:

1. Un cliente Windows se conecta a un servidor SAMBA que comparte recursos.
2. El cliente envía una solicitud de conexión al servidor SAMBA, que incluye el nombre del recurso compartido y las credenciales de autenticación.
3. El servidor SAMBA verifica las credenciales y, si son válidas, devuelve un identificador de sesión al cliente.
4. El cliente utiliza el identificador de sesión para acceder al recurso compartido y realizar operaciones.

## Kerberos

Kerberos es un protocolo de autenticación que utiliza un enfoque de "boleto" para verificar la identidad de los usuarios y los servicios. Fue desarrollado en el MIT y se utiliza en muchos entornos, incluyendo Windows y Unix.

Aquí hay un resumen de cómo funciona Kerberos:

1. Un cliente (por ejemplo, un usuario) se conecta a un servidor que utiliza Kerberos para autenticación.
2. El cliente solicita un boleto de autenticación al servidor Kerberos, que incluye el nombre de usuario y la contraseña.
3. El servidor Kerberos verifica las credenciales y, si son válidas, devuelve un boleto de autenticación al cliente.
4. El cliente utiliza el boleto de autenticación para acceder a los recursos del servidor sin necesidad de volver a autenticarse.

## WinRM

WinRM es un protocolo de gestión remota que permite a los administradores acceder y gestionar equipos Windows de forma remota. Utiliza el protocolo WS-Management (Web Services-Management) para comunicarse con los equipos remotos.

Aquí hay un resumen de cómo funciona WinRM:

1. Un administrador se conecta a un equipo remoto utilizando WinRM.
2. El administrador envía una solicitud de conexión al equipo remoto, que incluye las credenciales de autenticación.
3. El equipo remoto verifica las credenciales y, si son válidas, devuelve un identificador de sesión al administrador.
4. El administrador utiliza el identificador de sesión para acceder y gestionar el equipo remoto.

## RPC (Remote Procedure Call)

RPC es un protocolo que permite a los programas comunicarse con otros programas en diferentes equipos de una red. Permite a los programas llamar a procedimientos (funciones) en otros programas como si estuvieran en el mismo equipo.

Aquí hay un resumen de cómo funciona RPC:

1. Un programa (el cliente) se conecta a otro programa (el servidor) que ofrece un servicio RPC.
2. El cliente envía una solicitud de llamada al procedimiento al servidor, que incluye los parámetros de la llamada.
3. El servidor ejecuta el procedimiento y devuelve el resultado al cliente.
4. El cliente recibe el resultado y continúa ejecutando su programa.
