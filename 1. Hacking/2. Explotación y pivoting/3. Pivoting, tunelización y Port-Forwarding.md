## Direccionamiento IP y NIC

Cada computadora que se comunica en una red necesita una dirección IP. Si no tiene uno, no está en una red. La dirección IP se asigna en el software y, por lo general, se obtiene automáticamente de un servidor DHCP. También es común ver computadoras con direcciones IP asignadas estáticamente. La asignación de IP estática es común con:

- Servidores
- Routers
- Conmutar interfaces virtuales
- Impresoras
- Y cualquier dispositivo que proporcione servicios críticos a la red

Ya sea asignada o , la dirección IP se asigna a un (). Comúnmente, la NIC se conoce como o . Un ordenador puede tener varias NIC (físicas y virtuales), lo que significa que puede tener varias direcciones IP asignadas, lo que le permite comunicarse en varias redes. La identificación de oportunidades de pivote a menudo dependerá de las direcciones IP específicas asignadas a los hosts que comprometemos, ya que pueden indicar las redes a las que pueden llegar los hosts comprometidos. Es por eso que es importante para nosotros siempre verificar si hay NIC adicionales usando comandos como (en macOS y Linux) y (en Windows).`dynamically``statically``Network Interface Controller``NIC``Network Interface Card``Network Adapter``ifconfig``ipconfig`

## Enrutamiento

Es común pensar en un aparato de red que nos conecta a Internet cuando se piensa en un router, pero técnicamente cualquier ordenador puede convertirse en router y participar en el enrutamiento. Algunos de los desafíos a los que nos enfrentaremos en este módulo requieren que hagamos un host pivote que dirija el tráfico a otra red. Una forma de ver esto es a través del uso de AutoRoute, que permite que nuestra caja de ataque tenga que apuntar a redes que son accesibles a través de un host pivote. Una característica definitoria clave de un router es que tiene una tabla de enrutamiento que utiliza para reenviar el tráfico en función de la dirección IP de destino. Veamos esto en Pwnbox usando los comandos o .`routes``netstat -r``ip route`

Notaremos que Pwnbox, distribuciones de Linux, Windows y muchos otros sistemas operativos tienen una tabla de enrutamiento para ayudar al sistema a tomar decisiones de enrutamiento. Cuando se crea un paquete y tiene un destino antes de salir del equipo, la tabla de enrutamiento se utiliza para decidir dónde enviarlo. Por ejemplo, si estamos tratando de conectarnos a un objetivo con la IP, podríamos saber a partir de la tabla de enrutamiento a dónde se enviaría el paquete para llegar allí. # Redireccionamiento de Socat con un shell inverso

[Socat](https://linux.die.net/man/1/socat) es una herramienta de relé bidireccional que puede crear zócalos de tubería entre canales de red independientes sin necesidad de utilizar el túnel SSH. Actúa como un redirector que puede escuchar en un host y puerto y reenviar esos datos a otra dirección IP y puerto. Podemos iniciar el listener de Metasploit usando el mismo comando mencionado en la última sección en nuestro host de ataque, y podemos comenzar en el servidor Ubuntu.`2``socat`

#### Iniciar Socat Listener

  Redireccionamiento de Socat con un shell inverso


# Tunelización de RDP y SOCKS con SocksOverRDP

---

A menudo hay ocasiones durante una evaluación en las que podemos estar limitados a una red de Windows y es posible que no podamos usar SSH para pivotar. Tendríamos que utilizar herramientas disponibles para los sistemas operativos Windows en estos casos. [SocksOverRDP](https://github.com/nccgroup/SocksOverRDP) es un ejemplo de una herramienta que usa () de la característica Servicio de escritorio remoto de Windows. DVC es responsable de tunelizar los paquetes a través de la conexión RDP. Algunos ejemplos de uso de esta función serían la transferencia de datos del portapapeles y el uso compartido de audio. Sin embargo, esta función también se puede utilizar para tunelizar paquetes arbitrarios a través de la red. Podemos usarlo para tunelizar nuestros paquetes personalizados y luego hacer proxy a través de él. Utilizaremos la herramienta [Proxifier](https://www.proxifier.com/) como nuestro servidor proxy.`Dynamic Virtual Channels``DVC``SocksOverRDP`

> Using `netsh` to forward traffic from windows server to windows 10 workstation for `vfrank` user. [Port Forwarding with Windows Netsh](https://academy.hackthebox.com/module/158/section/1435)  
> In command prompt on the windows server `172.16.5.35`, create a `netsh` port forwarding proxy.

```
netsh.exe interface portproxy add v4tov4 listenport=8080 listenaddress=172.16.5.35 connectport=3389 connectaddress=172.16.6.25
```

> On `172.16.5.35` the listen port `8080` will direct traffic to port `3389` for destination target IP address `172.16.6.25`.

```
proxychains xfreerdp /v:172.16.5.35:8080 /u:vfrank /p:'Imply wet Unmasked!' /d:INLANEFREIGHT.LOCAL /dynamic-resolution /cert:ignore /drive:kali-drive,/home/kali/Downloads/htb/academy/webadmin
```
