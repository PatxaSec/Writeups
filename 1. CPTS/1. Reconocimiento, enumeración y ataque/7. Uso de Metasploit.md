# Introducción a Metasploit

---

Se trata de una plataforma modular de pruebas de penetración basada en Ruby que permite escribir, probar y ejecutar el código del exploit. Este código de exploit puede ser personalizado por el usuario o tomado de una base de datos que contiene los últimos exploits ya descubiertos y modularizados. Incluye un conjunto de herramientas que puede usar para probar vulnerabilidades de seguridad, enumerar redes, ejecutar ataques y evadir la detección. En esencia, es una colección de herramientas de uso común que proporcionan un entorno completo para las pruebas de penetración y el desarrollo de exploits.`Metasploit Project``Metasploit Framework``Metasploit Project`

![IMG](https://academy.hackthebox.com/storage/modules/39/S02_SS01.png)

Los mencionados son pruebas de concepto de exploits reales que ya se han desarrollado y probado en la naturaleza y se han integrado dentro del marco para proporcionar a los pentesters facilidad de acceso a diferentes vectores de ataque para diferentes plataformas y servicios. Metasploit no es un todoterreno, sino una navaja suiza con las herramientas suficientes para atravesar las vulnerabilidades sin parches más comunes.`modules`

Su punto fuerte es que proporciona una gran cantidad de objetivos y versiones disponibles, todos a unos pocos comandos de distancia de un punto de apoyo exitoso. Estos, combinados con un exploit hecho a medida para esas versiones vulnerables y con una carga útil que se envía después del exploit, que nos dará acceso real al sistema, nos proporcionan una forma fácil y automatizada de cambiar entre las conexiones objetivo durante nuestras aventuras posteriores a la explotación.

# Módulos

---

Como mencionamos anteriormente, Metasploit son scripts preparados con un propósito específico y las funciones correspondientes que ya han sido desarrollados y probados en la naturaleza. La categoría consiste en la llamada prueba de concepto () que se puede utilizar para explotar las vulnerabilidades existentes de una manera en gran medida automatizada. Mucha gente suele pensar que el fracaso del exploit refuta la existencia de la supuesta vulnerabilidad. Sin embargo, esto es solo una prueba de que el exploit Metasploit no funciona y no de que la vulnerabilidad no exista. Esto se debe a que muchos exploits requieren personalización de acuerdo con los hosts de destino para que el exploit funcione. Por lo tanto, las herramientas automatizadas como el framework Metasploit sólo deben considerarse una herramienta de apoyo y no un sustituto de nuestras habilidades manuales.`modules``exploit``POCs`

#### Nombre

Finalmente, la etiqueta explica la acción real que se puede realizar utilizando este módulo creado para un propósito específico.`Name`

## Tipos de destino

Hay una gran variedad de tipos de objetivos. Cada objetivo puede variar de otro según el paquete de servicio, la versión del sistema operativo e incluso la versión del idioma. Todo depende de la dirección de retorno y otros parámetros en el objetivo o dentro del módulo de explotación.

La dirección de retorno puede variar porque un paquete de idioma en particular cambia de dirección, hay una versión de software diferente disponible o las direcciones se cambian debido a ganchos. Todo está determinado por el tipo de dirección de retorno requerida para identificar el objetivo. Esta dirección puede ser , un salto a un registro específico que identifica al objetivo, o un archivo . Para obtener más información sobre el tema de las direcciones de retorno, consulte el módulo [Desbordamientos de búfer basados en pila en Windows x86](https://academy.hackthebox.com/module/89/section/931). Los comentarios en el código del módulo de exploit pueden ayudarnos a determinar cuál es el objetivo.`jmp esp``pop/pop/ret`

Para identificar un objetivo correctamente, necesitaremos:

- Obtener una copia de los archivos binarios de destino
- Utilice msfpescan para localizar una dirección de devolución adecuada

Más adelante en el módulo, profundizaremos en el desarrollo de exploits, la generación de carga útil y la identificación de objetivos.

#### Individuales

Una carga útil contiene el exploit y todo el shellcode de la tarea seleccionada. Las cargas útiles en línea son, por diseño, más estables que sus contrapartes porque contienen todo en uno. Sin embargo, algunos exploits no soportarán el tamaño resultante de estas cargas útiles, ya que pueden llegar a ser bastante grandes. son cargas útiles autónomas. Son el único objeto enviado y ejecutado en el sistema de destino, obteniéndonos un resultado inmediatamente después de su ejecución. Una sola carga útil puede ser tan simple como agregar un usuario al sistema de destino o iniciar un proceso.`Single``Singles`

#### Escenógrafos

`Stager` Las cargas útiles funcionan con las cargas útiles de etapa para realizar una tarea específica. Un Stager está esperando en la máquina atacante, listo para establecer una conexión con el host víctima una vez que el escenario complete su ejecución en el host remoto. Por lo general, se utilizan para establecer una conexión de red entre el atacante y la víctima, y están diseñados para ser pequeños y confiables. Metasploit usará el mejor y recurrirá a uno menos preferido cuando sea necesario.`Stagers`

Etapas de Windows NX frente a NO-NX

- Los stagers NX son más grandes (memoria VirtualAlloc)
- El valor predeterminado ahora es compatible con NX + Win7

#### Etapas

`Stages` son componentes de carga útil que son descargados por los módulos de stager. Las diversas etapas de carga útil proporcionan funciones avanzadas sin límites de tamaño, como Meterpreter, inyección VNC y otras. Las etapas de carga útil utilizan automáticamente etapas intermedias:

- Un solo error con cargas útiles grandes`recv()`
- El Stager recibe el stager intermedio
- A continuación, el Stager intermedio realiza una descarga completa
- También es mejor para RWX

#### Carga útil de Meterpreter

La carga útil es un tipo específico de carga útil multifacética que se utiliza para garantizar que la conexión con el host víctima sea estable, difícil de detectar mediante comprobaciones simples y persistente en todos los reinicios o cambios del sistema. Meterpreter reside completamente en la memoria del host remoto y no deja rastros en el disco duro, lo que lo hace muy difícil de detectar con técnicas forenses convencionales. Además, los scripts y los complementos se pueden mover dinámicamente según sea necesario.`Meterpreter``DLL injection``loaded and unloaded`

Una vez que se ejecuta la carga útil de Meterpreter, se crea una nueva sesión, que genera la interfaz de Meterpreter. Es muy similar a la interfaz msfconsole, pero todos los comandos disponibles están dirigidos al sistema de destino, al que la carga útil ha "infectado". Nos ofrece una gran cantidad de comandos útiles, que van desde la captura de pulsaciones de teclas, la recopilación de hash de contraseñas, la pulsación del micrófono y la captura de pantalla hasta la suplantación de tokens de seguridad de procesos. Profundizaremos en más detalles sobre Meterpreter en una sección posterior.

Usando Meterpreter, también podemos en diferentes Plugins para ayudarnos con nuestra evaluación. Hablaremos más sobre esto en la sección Plugins de este módulo.`load`

## Uso de la base de datos

Con la ayuda de la base de datos, podemos administrar muchas categorías y hosts diferentes que hemos analizado. Después de confirmar que la base de datos está conectada correctamente, podemos organizar nuestro archivo .`Workspaces`

#### Espacios de trabajo

Podemos pensar de la misma manera que pensaríamos en las carpetas de un proyecto. Podemos segregar los diferentes resultados de escaneo, hosts e información extraída por IP, subred, red o dominio.`Workspaces`

Para ver la lista actual del área de trabajo, utilice el comando. Al agregar un modificador o después del comando, seguido del nombre del área de trabajo, se incorporará una o esa área de trabajo a la base de datos.`workspace``-a``-d``add``delete`

## Mezclas

El Metasploit Framework está escrito en Ruby, un lenguaje de programación orientado a objetos. Esto juega un papel importante en lo que hace que su uso sea excelente. Los mixins son una de esas características que, cuando se implementan, ofrecen una gran cantidad de flexibilidad tanto para el creador del script como para el usuario.`msfconsole`

Los mixins son clases que actúan como métodos para su uso por otras clases sin tener que ser la clase principal de esas otras clases. Por lo tanto, se consideraría inapropiado llamarlo herencia, sino más bien inclusión. Se utilizan principalmente cuando:

1. Desea proporcionar muchas características opcionales para una clase.
2. Desea utilizar una característica en particular para una multitud de clases.

La mayor parte del lenguaje de programación Ruby gira en torno a Mixins como módulos. El concepto de Mixins se implementa utilizando la palabra , a la que le pasamos el nombre del módulo como un . Podemos leer más sobre mixins [aquí](https://en.wikibooks.org/wiki/Metasploit/UsingMixins).`include``parameter`

Si recién estamos comenzando con Metasploit, no debemos preocuparnos por el uso de Mixins o su impacto en nuestra evaluación. Sin embargo, se mencionan aquí como una nota de lo compleja que puede llegar a ser la personalización de Metasploit.
